SYN 攻击及其防范措施 🚨

什么是 SYN 攻击？ ⚠️

SYN 攻击 是一种常见的 拒绝服务攻击（DoS），攻击者利用 TCP 连接建立过程中的三次握手漏洞，通过大量伪造的 SYN 请求报文来消耗服务器资源。

攻击过程：

  1. 伪造源 IP 地址
    攻击者伪造大量 不同的源 IP 地址，向目标服务器发送 SYN 请求报文。

  2. 服务器响应
    服务器在接收到这些 SYN 请求后，会生成 SYN-ACK 报文并将其发送回攻击者（实际上，返回的 SYN-ACK 报文并不会得到回应，因为源 IP 是伪造的）。

  3. 半连接队列耗尽
    服务器的 半连接队列 用来存储等待三次握手完成的连接请求。当攻击者发送大量 SYN 请求时，服务器的半连接队列被填满，导致后续合法的连接请求被丢弃。最终，服务器无法正常处理来自真实用户的连接请求。

半连接队列与全连接队列：
  • 半连接队列：存储尚未完成三次握手的连接（即接收到 SYN 请求，但未收到 ACK 响应）。
  • 全连接队列：存储已完成三次握手的连接，准备进行数据传输。

如何避免 SYN 攻击？ 🛡️

  1. 增加 TCP 半连接队列的大小 🔒
    通过增大半连接队列的大小，服务器能够处理更多的连接请求。虽然这能一定程度上缓解攻击，但并不能从根本上解决问题，因为攻击者可以通过伪造更多的 SYN 请求来继续占满队列。

  2. 减少 SYN + ACK 重传次数 🔄
    在 SYN 攻击过程中，服务器常常无法收到攻击者的 ACK 报文，因此会持续重传 SYN-ACK 报文。通过调整 tcp_synack_retries 参数来减少重传次数，可以减少服务器的负担。

    • tcp_synack_retries：控制服务器在没有收到 ACK 报文时，重新发送 SYN-ACK 的次数。默认情况下，通常是 5 次，可以适当减少此值，从而加快连接的断开速度，降低对服务器的资源消耗。

  3. 启用 SYN Cookies 🧩
    SYN Cookies 是一种防止 SYN 攻击的技术。它通过在服务端生成一个 加密的 cookie（而不是将连接信息存储在内存中）来替代将连接请求存储在半连接队列中的做法。当客户端回应 SYN-ACK 时，服务端验证该 cookie 来确认连接请求的合法性，从而防止无效的 SYN 请求占用系统资源。

  4. 防火墙过滤 🔥
    使用防火墙或专门的入侵检测系统（IDS）来过滤 伪造的源 IP 地址，或者对来自相同源 IP 地址的连接请求进行速率限制，从而减缓 SYN 攻击的影响。

总结： 📋

  • SYN 攻击 通过伪造大量的 SYN 请求占用服务器的半连接队列，导致服务不可用。
  • 防范措施 包括：增加半连接队列大小、减少 SYN-ACK 重传次数、启用 SYN Cookies 和使用防火墙过滤伪造 IP 地址等。

这些防范措施可以有效缓解或避免 SYN 攻击，确保服务器资源不被恶意耗尽，从而保持服务的可用性和稳定性。
